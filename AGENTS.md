# ガイドライン

## レポジトリ

- GitHubのf0reachARR/wifimanに置かれています

## コマンド

- pnpmを使用して依存関係を管理してください。

## TypeScriptの使用

- TypeScriptのstrictモードを有効にしてください。
- 可能な限り型注釈を使用してください。
- シングルクォートとセミコロンを使用してください。
- コードの品質を保つために、Biomeを使用してコードをフォーマットし、tscを使用して型チェックを行ってください。

## Drizzleのマイグレーション作成

Drizzleのマイグレーションを作成する際は、以下の手順に従ってください。

- docker compose up -d postgres を実行してPostgreSQLコンテナが起動していることを確認してください。
- pnpm db:generate コマンドを実行して、マイグレーションファイルを生成してください。
- 生成されたマイグレーションファイル(./apps/api/drizzle以下にあるはずです)を確認し、必要に応じて編集してください。
- pnpm db:migrate コマンドを実行して、マイグレーションを適用してください。

## 一時ファイル

- 一時ファイルは .github/tmp ディレクトリに保存してください。

## Codex サブエージェント運用

このリポジトリでユーザーから機能追加、バグ修正、リファクタリング、調査から実装までを含むタスクを依頼された場合は、デフォルトで Codex のサブエージェントを使って進めてください。

### 標準フロー

1. まず `issue_manager` を呼び出し、ユーザーの依頼全文を渡してください。既存 Issue の重複確認、必要に応じた Issue 作成/更新、実装可能な単位への分割、Issue 番号・URL・スコープ・受け入れ条件の返却を依頼します。
2. 返却された Issue は、ユーザーが並列実行を明示しない限り 1 件ずつ直列に進めてください。
3. 各 Issue について `implementation_planner` を呼び出し、Issue 番号、URL、本文、ユーザー文脈を渡してください。ブランチ名、変更対象、公開インターフェース、テスト方針、検証コマンドを含む、実装者が判断に迷わない計画を依頼します。
4. 計画ができたら `tdd_implementer` を呼び出し、Issue と計画を渡してください。Issue 番号を含むブランチ作成/切り替え、TDD に沿った実装、検証、コミットまでを依頼します。
5. 実装後は `implementation_reviewer` を呼び出し、Issue、計画、差分概要、コミット、検証結果を渡してください。レビュー結果には必ず `PR_READY` または `CHANGES_REQUESTED` のどちらかを含めるよう依頼します。
6. `CHANGES_REQUESTED` の場合は、指摘内容を `tdd_implementer` に戻して修正を依頼し、再度 `implementation_reviewer` に確認させてください。このサイクルは `PR_READY` になるか、ブロッカーが明確になるまで続けます。
7. `PR_READY` になったら `pr_author` を呼び出し、Issue、ブランチ、コミット、検証結果、PR に含めるべき注意事項を渡してください。PR 作成、Issue への `Closes #<number>` の付与、PR URL の返却を依頼します。
8. 最後に、Issue 番号、ブランチ、コミット、実行した検証、レビュー結果、PR URL をユーザーへ簡潔に報告してください。

### 委任時のルール

- 親エージェントは、実装やレビューを自分で抱え込まず、適切なサブエージェントへ完全な文脈を渡してください。
- 実装とレビューの責務は分離してください。レビューが `PR_READY` を返すまで PR を作成してはいけません。
- 必要な承認、GitHub 認証、ネットワークアクセス、権限が不足している場合は、失敗したコマンドや必要な対応を明確に報告してください。
- 関係のないユーザー変更は戻さないでください。対象ファイルに既存変更がある場合は、それを前提に作業してください。
